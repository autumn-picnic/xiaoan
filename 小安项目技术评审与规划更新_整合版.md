## TL;DR — 执行摘要

本次技术会议完成了对小安聊天机器人现有系统的深度审查，识别出三个核心问题：关键词触发机制存在语义盲区、大多数胶囊的图谱检索层为空、评测框架未覆盖真实用户场景的关键边界条件。

完成了对7个情感/心理类数据集和4个中文法律类基准的调研，并确定核心决策：优先采集30-50条真实（脱敏）家暴咨询对话作为后续提示词优化的训练锚点，而非依赖合成数据。

三阶段路线图已确认：近期（6-8周）合并评测框架，中期（2-3个月）升级检索层，长期（3-6个月）引入自动化提示词优化闭环。

## 一、当前系统状态

### 1.1 处理管道

```
safety.scan → router → should_load_ground → compose.llm → output
```

- **should_load_ground** (render_policy.py)：三层判断——胶囊策略覆盖 → 会话粘性 → 22个关键词子串匹配
- **compose.llm** (compose.py:148)：系统提示词硬编码，温度固定0.3，向LLM发送user_message + capsule recognize/act/ground + resolved_ground JSON载荷

### 1.2 已确认的关键缺陷

| 问题 | 表现 | 根因 |
|------|------|------|
| 触发语义盲区 | "我能申请保护令吗"不触发检索，"我怎么申请保护令"才触发 | 纯关键词子串匹配，无语义理解 |
| 图谱层为空 | K1等胶囊 ground.nodes=[] | 法律内容依赖字符串字段，未链接真实wiki节点 |
| 法律胶囊策略不足 | n5p/n3a/n2a 设为 on_demand 而非 always | 配置问题 |
| 8行回退窗口 | ground.py:225 硬编码，法律条文常超8行 | 缺乏可配置化 |

### 1.3 现有评测框架

- **Level 0**（已有）：20条单轮路由冒烟测试，路由准确率100%
- **Level 1**（FindBestBaseModel）：TC-01~TC-13，7维度LLM-judge评分，haiku用户模拟器

### 1.4 should_load_ground 逻辑详解

三层判断机制，优先级从高到低：

1. **静态策略覆盖** (per-capsule render_policy.ground)
   - `always` / `on_entry` → 直接加载
   - `never` → 永不加载
   - `on_demand` → 继续往下判断（默认值）

2. **会话粘性** (already_loaded=True)
   - 本次对话中一旦触发过，后续轮次保持加载

3. **关键词子串匹配** (has_ground_intent)
   - message中包含22个默认触发词之一，或胶囊自定义的ground_triggers
   - 触发词包括："证据"、"怎么申请"、"法律依据"、"能不能"、"可以吗"等

4. **默认行为**: on_demand + 无匹配 → 不加载

**关键发现**：纯字面子串匹配，无语义理解。"我能申请保护令吗"里没有触发词不会加载ground，但"我怎么申请保护令"（含"怎么申请"）会加载。这是最主要的调优空间。

## 三、关键决策

**决策：选择 Path 2 — 真实数据优先**

**问题**：DSPy MIPRO可以自动优化提示词，但若以haiku模拟器生成的合成对话为训练锚点，优化结果将对真实用户产生分布偏移。家暴求助者的语言模式（不连贯、情绪化、反复确认）与合成数据存在本质差异，在高风险场景中此偏移不可接受。

**决定**：优先采集30-50条真实脱敏家暴咨询对话，参照CPsyCoun的报告重建方法，作为DSPy优化的训练数据。在此之前不启动提示词自动优化工作。

## 七、前沿替代方案（长期考虑）

### Self-RAG架构对比

| 维度 | 当前chatflow | Self-RAG替代 |
|------|-------------|-------------|
| 触发检索 | 关键词匹配 | 模型生成 [Retrieve] token |
| 检索质量评估 | 无 | 模型生成 [ISREL] 相关性评分 |
| 生成可信度 | 无 | 模型生成 [ISSUP] 支持度评分 |
| 迭代检索 | 无 | 不够时自动再检索 |

Self-RAG的核心论文：Self-RAG: Learning to Retrieve, Generate, and Critique。对XiaoAn最直接的价值：把 should_load_ground 的keyword逻辑换成模型自己判断"我需要法律依据吗"。

### LangGraph式多步Agentic RAG

把现在的线性pipeline改为：

```
用户消息
  ↓
[路由agent] → 判断capsule
  ↓
[检索决策agent] → 判断需要哪些知识节点
  ↓
[检索agent] → 从知识图谱取ground items
  ↓  ← 如果相关性不足，回到检索决策
[生成agent] → compose回答
  ↓
[验证agent] → 检查是否满足安全红线 + 内容质量
  ↓  ← 如果不满足，regenerate
[输出]
```

这就是 Corrective RAG (CRAG) + LangGraph 的组合。对XiaoAn的价值是可以在生成前自我验证法律准确性。

### Agentic RAG 引入策略（2026-07-13 更新）

#### 什么是 Agentic RAG

Standard RAG 是固定管道（query → retrieve → generate）；Agentic RAG 把 LLM 升格为 orchestrator——它自己决定要不要再检索、再评估、走哪条路，直到认为答案足够好。代价：每次决策 = 额外的 LLM call = 延迟 + 成本 + 不确定性。

#### 各 Agentic 行为对 XiaoAn 的适合度

| Agentic RAG 行为 | 适合度 | 原因 |
|---|---|---|
| 查询分解（把"我想离婚又要保护令"拆成子问题）| 高 | 跨胶囊需求真实存在 |
| 迭代图遍历（沿边自动找相关节点）| 高 | edges.md 结构已有，等待被利用 |
| 自我反思（"我的回答够好吗？"再检索）| 中 | 复杂法律查询有用，危机场景不能等 |
| 自主决定是否触发 Crisis SOP | **禁止** | 安全判断不能委托给 Agent |
| 自主决定 PII 脱敏粒度 | **禁止** | 隐私边界不能动态协商 |

#### 成本效益

**延迟成本（最关键）**：Agentic RAG 的自我反思循环每轮至少多 2-4 秒。反思 2-3 轮总延迟可达 10-15 秒。对 DV 用户，10 秒无回应等同于系统沉默，可能直接关闭。

**金钱成本**：

| 模式 | 每 turn LLM calls（估算）|
|---|---|
| 现有 fixed pipeline | 2 calls（router + support agent）|
| 加入 bounded agentic ground resolver | 2-3 calls |
| 完整 Agentic RAG with self-reflection | 4-8 calls |

Full agentic ≈ 成本乘以 3-4x。对公益产品不可持续。

#### MVP 直接用的 Pros/Cons

**Pros**：
- baseline 路径有图谱支撑而非 LLM 靠自身知识回答
- 跨胶囊复杂问题（离婚 + 保护令 + 孩子）可更完整回答
- 图遍历能自动发现"作者忘记写的 locator"——系统更健壮

**Cons**：
- Full Agentic RAG 在 MVP 几乎没有足够 eval——agent 行为路径指数级，无法确保危机场景不走错路
- 调试困难：中间推理步骤不透明，出错时难定位
- edges.md 尚未被解析成 runtime 可遍历的数据结构，是必要的前置工程

**结论：MVP 不建议做 full Agentic RAG，但可以做"有边界的局部 Agentic"。**

#### 带边界的 Agentic RAG Chatflow（推荐方案）

```
User Input
    │
    ▼
[PII Redaction Gate]              ← 永远 FIXED，不 agentify
    │
    ▼
[Safety Scan]                     ← 永远 FIXED，不 agentify
    │
  red flag → [Crisis SOP]         ← 永远 FIXED
    │
  normal
    │
    ▼
[Intent Classifier]
    │
    ├── 具体意图（能命中胶囊）
    │       │
    │       ▼
    │   [Capsule Router]           ← FIXED（LLM-as-router 保持）
    │       │
    │       ▼
    │   ★[Bounded Graph Agent]★  ← MVP 可做的 agentic 部分
    │   1. 从胶囊 locators 出发
    │   2. 查 edges.md，沿 is_element_of/proves/enables 展开 1-2 跳
    │   3. 一次自我评估："节点够不够回答问题？"
    │   4. 硬限制：max 3 retrieval steps，max 6 nodes（不超过一次反思）
    │       │
    │       ▼
    │   [Context Assembly + Main Support Agent]  ← 单次 LLM call
    │
    └── 开放/全局查询（无胶囊命中）
            │
            ▼
        ★[Graph Planning Agent]★  ← Production 阶段实现
        1. 把问题拆解成 2-3 个子问题
        2. 每个子问题查询图谱节点
        3. 自我反思："哪部分还没有答案？"
        4. 补充查询（max 2 轮）
        5. 聚合生成 Summary
        硬限制：max 4 steps，max 10 nodes，max 20s total
            │
            ▼
        [Main Support Agent]      ← 单次 LLM call
    ↓（两条路径 merge）
[Output Guard]                    ← FIXED
    │
    ▼
[Final Response]
    + Signpost hint（相邻胶囊导航，1句，见 §十）
```

#### 分阶段引入建议

| 阶段 | 工作内容 | 说明 |
|---|---|---|
| **MVP（当前）** | 把 edges.md 解析成 Python dict；Ground Resolver 命中 node 后自动拉入 1-hop `proves`/`is_element_of` 邻居 | 无 LLM 决策循环，近零额外延迟，效果接近图遍历 |
| **Post-MVP** | Bounded Graph Agent 加单次自我评估（Yes/No："内容足够吗？"）；只有 No 才触发一次补充查询 | 每 turn 最多多 1 次 LLM call |
| **Production** | Graph Planning Agent 用于开放查询路径；此时有足够 eval 数据验证 agent 行为安全边界 | 多轮反思，需严格的混淆矩阵和安全测试 |

**关于"自我完善"**：Agentic RAG 让查询时更智能，但不等于系统自我完善。"自我完善"需要另一个机制：把 agent 找不到答案的 query 记录下来，定期人工审查，转化成新胶囊或新 wiki 节点。这是 `knowledge_strategy.md` §2 的"数据飞轮驱动的内容建设"——Agentic RAG 是轮子，飞轮的燃料还是人的判断。

## 九、知识架构与 GraphRAG 对齐分析（2026-07-13）

### 9.1 核心概念对照

Microsoft GraphRAG 的三个核心创新（相比普通 RAG）：

1. **索引期建图**：用 LLM 从文档自动抽取实体 + 关系，形成知识图谱（而非向量化原文）
2. **社区检测 + 预生成摘要**：把图分成多层次"社区"，提前为每个社区生成摘要（等同于查询前的语义压缩）
3. **双模式查询**：Local Search（从命中实体出发沿边遍历）vs Global Search（跨社区聚合，回答全局性高层问题）

普通 RAG 的失败模式正是本项目 `knowledge_strategy.md` §2 所指——直接丢法条给大模型，要求它同时扮演法官、律师、心理咨询师，"顾此失彼"。

### 9.2 当前系统 vs GraphRAG 的结构对应

| GraphRAG 概念 | 小安系统对应 | 评估 |
|---|---|---|
| 文档语料库 | `knowledge/source/` 原始法条 | 完全对应 |
| LLM 自动抽取实体图 | **人工策划** legal atom nodes | **更优**（法律语义需精确，自动抽取会出错）|
| 知识图的边/关系 | `edges.md` 的 12 种 relation types | 完全对应，语义比 GraphRAG 更精确 |
| 社区检测 + 多层次层级 | Tier 1–5 信任层 + F1–F5 认识论功能 | 概念对应，但**静态定义**而非算法动态聚类 |
| 社区预摘要（Community Reports）| **场景胶囊**（Capsules，R-A-G 槽位） | **完全对应**，胶囊就是人工策划的 community report |
| Local Search | Capsule Router → Ground Resolver 路径 | 部分对应（见 Gap 分析）|
| Global Search | **当前缺失** | 最大架构空白 |

**结论**：小安做的是"领域专用、人工策划版 GraphRAG"，这对法律场景比自动抽取更适合。三层架构（source → wiki/graph → capsules）在 GraphRAG 框架下完全合理。

### 9.3 三个架构 Gap

**Gap 1：Ground Resolver 是静态查找，不是图遍历**

GraphRAG Local Search 的核心是沿边展开：命中一个节点后，顺着 `is_element_of`、`enables`、`proves` 等边自动拉入相邻节点内容。

当前 Ground Resolver 只做：`locator: "node:personal-safety-protection-order"` → 返回该节点原子结论。

它**没有利用** edges.md 里已有的边关系。理想行为：
```
personal-safety-protection-order
  ← is_element_of ← protection-order-element-danger  (自动拉入)
  ← proves ← protection-order-evidence               (自动拉入)
  ← provides_evidence_for ← public-security-response-duty (可选拉入)
```

当前行为：胶囊作者需要手动在 `basis` 里写每个 locator，容易漏掉邻居节点。

**Gap 2：Global Search 完全缺失**

对应以下类型用户问题（当前全部 fallback 到无图谱支撑的 Main Agent）：
- "家暴受害者有哪些法律保护路径？"（横跨多个 capsule）
- "我不知道自己需要什么帮助"（无法匹配任何 capsule）
- "离婚和保护令我能同时申请吗？"（跨 N5d、N5p、N5e 三个胶囊）

**Gap 3：Tier 5 学术层只在 NotebookLM，未接入运行时**

edges.md 已有 `deepens`、`contextualizes_reality_of` 等边，但 NotebookLM 是离线查询工具。用户问"为什么警察说家务事不管"时，`state-intervention-limits` 节点无法自动注入。

### 9.4 对齐方案（分阶段）

**MVP（当前可做）**：把 edges.md 解析成 Python dict，Ground Resolver 在命中 node 后自动拉入 1-hop 的 `proves` 和 `is_element_of` 邻居节点。不需要向量数据库，几乎零额外延迟。

**Post-MVP**：为"无膠囊命中的开放查询"建 Global Search 路径。最简实现：把 `legal-mechanism-tree.md` + 所有 capsule 的 `title + triggers + recognize` 合成一个"知识图摘要"文档，baseline 路由时注入 LLM，替代当前完全不给图谱信息的做法。

**Production**：Tier 5 concept-depth 节点的核心结论 inline 进节点文件，Ground Resolver 命中 `deepens` 边时能直接读取，无需查 NotebookLM。

## 十、单胶囊设计原则与 Signpost 机制（2026-07-13）

### 10.1 单胶囊设计的合理性

单次回应只激活一个胶囊的设计，在家暴场景下**基本正确**，依据：

- **认知负担**：`knowledge_strategy.md` §1 明确指出"用户认知带宽几近于零"，信息过载等于无效帮助
- **创伤知情设计**：DV 支持组织（National DV Hotline 等）的接线员训练准则：每次通话只推进一个可执行步骤，让幸存者掌控节奏
- **已有机制支撑**：`active_capsule_id` + TTL 的对话连续性保证"用户可以继续追问"是可行路径，不是空话

### 10.2 核心漏洞：「不知道自己不知道」

单胶囊设计有一个前提假设：**用户知道有什么可以问**。在 DV 场景下，这个假设经常失效。

典型场景：用户问"警察不管怎么办" → 系统命中 N3a 推诿应对。她可能完全不知道"人身安全保护令"的存在，不知道的东西不会问，永远发现不了 N5p。

这不是多胶囊 vs 单胶囊的问题，而是**主动导航缺失**。

### 10.3 Signpost 机制（轻量路标）

**设计**：每个胶囊在 `act` 末尾或 `limits` 内，加一条"下一步可以问我的内容"指针。

示例（N3a 报警指引末尾）：
> "如果你还想了解如何申请让他不能靠近你，可以告诉我。"

**特点**：
- 不输出两个胶囊，不增加认知负担
- 主动降低"不知道能问什么"的障碍
- 与 `do_not_use_when` 路由逻辑互补：router 知道当前胶囊的边界，signpost 告诉用户边界外还有什么
- GraphRAG 术语：等同于"related community surfacing"——主答案来自命中社区，顺带提示邻近社区

**实施**：在 capsule schema 中新增可选字段 `next_hints`，内容由胶囊作者维护，answer composer 在回答末尾轻量附加。

```yaml
# capsule schema 扩展
next_hints:                          # 可选，最多2条
  - "如果你也想了解申请保护令的步骤，可以告诉我"
  - "如果警察还是不处理，可以问我投诉渠道"
```

### 10.4 例外：Baseline 路径需要主动导航

"用户完全不知道从哪里开始"的开放性输入（对应§7 的 no-match/baseline 路由），单胶囊设计不适用——根本没有命中胶囊。

此时 Main Agent Baseline SOP 应主动列出 2-3 个高频起点供选择，而不是只做情绪支持。

### 10.5 评测补充建议

在 TC 库中加入一类"沉默披露"测试：用户只描述处境（"他又打我了"），不问任何具体问题。验证系统是否能在不强行推胶囊的前提下，自然打开对话空间，同时让用户知道可以获得具体帮助。对应 CounselBench 的接触维度测试。

## 附录：技术洞察汇总

### 关于PROTEA与Self-RAG

- PROTEA（2025年5月）是目前最接近需求的框架：它对multi-agent workflow的每个节点单独打分，自动定位瓶颈，生成修改建议，形成eval→改进→eval闭环。这正是目标B（系统调试）缺少的。
- Self-RAG的核心是让模型自己生成[Retrieve]/[IsRel]反思token——这比关键词触发更自然，但需要fine-tuning。CRAG不需要fine-tuning，直接加一个后检索的相关性评估器，是成本最低的RAG升级路径。
- DSPy MIPRO 和 TextGrad 解决的是不同粒度的问题：MIPRO优化整个system prompt（全局），TextGrad优化每个组件的prompt（局部，如compose.py的system部分和各capsule的act/recognize指令）。

### 关于系统架构

- compose.py:148的 system prompt 是内联硬编码在 build_messages 里，而不在任何可配置文件中——这是自动化prompt优化的第一个接入点。
- capsules.json 中 K1 的 ground.nodes = []，说明目前大多数capsule的图RAG层是空的，RAG还是靠 legal_basis 字符串而不是真实wiki节点支撑——这是最大的内容质量缺口。
- 整个pipeline是线性的：safety → router → should_load_ground → compose。这个结构可以直接被 Self-RAG 式的反思循环替代。

### 关于数据集方法论

- CPsyCoun用的是"report→dialogue reconstruction"两阶段法：先有真实咨询报告，再重建对话。这比直接合成对话质量高很多，因为对话的逻辑是从真实咨询案例倒推的——XiaoAn的TC扩充可以学这个思路。
- LawBench特别测"abstention rate"（模型说我不知道/不能做的比例），这对XiaoAn的"能力边界"维度是最直接的对标。

### 关于评测架构设计

- autoresearch的核心设计原理：锁eval harness，放开被调变量。val_bpb是固定靶，train.py是自由变量。这个模式直接可以套用：锁 test_cases + scorer，调RAG config + capsule prompt。
- 当前eval的最大盲点：只有最终答案质量，没有RAG层分离信号。无法区分"检索了错误的ground"还是"检索对了但生成时没用上"——这是调试RAG最关键的两步分解。
