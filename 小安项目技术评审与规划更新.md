  TL;DR — 执行摘要

  本次技术会议完成了对小安聊天机器人现有系统的深度审查，识别出三个核心问题：关键词触发机制存在语义盲区、大多数胶囊的图谱检索层为空、评测框架未覆盖真实用户场景的关键边界条件。

  完成了对7个情感/心理类数据集和4个中文法律类基准的调研，并确定核心决策：优先采集30-50条真实（脱敏）家暴咨询对话作为后续提示词优化的训练锚点，而非依赖合成数据。

  三阶段路线图已确认：近期（6-8周）合并评测框架，中期（2-3个月）升级检索层，长期（3-6个月）引入自动化提示词优化闭环。

  三、关键决策

  决策：选择 Path 2 — 真实数据优先

  问题：DSPy MIPRO可以自动优化提示词，但若以haiku模拟器生成的合成对话为训练锚点，
  优化结果将对真实用户产生分布偏移。家暴求助者的语言模式（不连贯、情绪化、反复确
  认）与合成数据存在本质差异，在高风险场景中此偏移不可接受。

  决定：优先采集30-50条真实脱敏家暴咨询对话，参照CPsyCoun的报告重建方法，作为DSPy
  优化的训练数据。在此之前不启动提示词自动优化工作。

   ★ Insight─────────────────────────────────────
  - PROTEA（2025年5月）是目前最接近你需求的框架：它对multi-agent
  workflow的每个节点单独打分，自动定位瓶颈，生成修改建议，形成eval→改进→eval闭环
  。这正是目标B（系统调试）缺少的。
  - Self-RAG的核心是让模型自己生成[Retrieve]/[IsRel]反思token——这比关键词触发更自
  然，但需要fine-tuning。CRAG不需要fine-tuning，直接加一个后检索的相关性评估器，
  是成本最低的RAG升级路径。
  - DSPy MIPRO 和 TextGrad 解决的是不同粒度的问题：MIPRO优化整个system
  prompt（全局），TextGrad优化每个组件的prompt（局部，如compose.py的system部分和
  各capsule的act/recognize指令）。

  做这个整合时有一个核心tension值得想清楚：

  你现在的7维度评分体系（基础能力、行动赋权等）是回答质量的评估，来自human-in-the
  -loop的咨询师视角。但你刚才说的目标是"调试chatflow、找最好的RAG调用方式"——这两
  个目标在评估粒度上是冲突的：

  - 调RAG需要的是组件级信号（检索了什么、用了多少），粒度很细，每轮都要记
  - 评回答质量需要的是对话级信号（整体7维度），粒度粗，每个TC出一个数

  如果直接用weighted_total作为RAG调优的loss，信号太稀疏——跑完一个TC-04（4轮）才能
  知道一个数字，但你不知道是第几轮的检索出了问题。

  - A) 模型选型（哪个base model更好）——
  现有7维度够用，对应FindBestBaseModel原始目的
  - B) 系统调试（调RAG触发策略、调ground节点、调system prompt）——
  需要per-turn的组件级日志 + 细粒度信号

  这两个目的需要不同的eval设计。目前看你说的是B，那架构重心应该是"可观测性"而非"
  综合评分"——有点像分布式系统里trace的思路，而不是benchmark的思路。你同意这个方向

  基于autoresearch + FlashRAG + BenchmarkQED，你的eval framework需要加三层：

  1. RAG组件分离指标（最关键的缺口）

  现在scorer.py的7维度都在评最终回答。加两个检索层指标（RAGAS做法）：

  ┌────────────────────┬─────────────────────────────────┬───────────────────┐
  │      新增指标      │              问题               │     判断方法      │
  ├────────────────────┼─────────────────────────────────┼───────────────────┤
  │ retrieval_relevanc │ 检索到的ground                  │ judge对比         │
  │ e                  │ node内容和用户问题相关吗？      │ (user_query,      │
  │                    │                                 │ retrieved_ground) │
  ├────────────────────┼─────────────────────────────────┼───────────────────┤
  │ answer_groundednes │ 最终回答是否实际引用了检索结果  │ judge对比 (retrie │
  │ s                  │ ？                              │ ved_ground,       │
  │                    │                                 │ assistant_reply)  │
  └────────────────────┴─────────────────────────────────┴───────────────────┘

  这两个指标加进scorer.py 的judge
  prompt里，就能告诉你：分低是"RAG取错了"还是"取对了但compose没用"。

  2. autoresearch式的实验循环（锁eval，调config）

  参考FlashRAG的YAML config模式，在poc/创建一个 experiment.yaml：

  # experiment.yaml —每次实验改这一个文件
  experiment_id: exp_001
  notes: "baseline: top_k=3, no rerank"

  rag:
    top_k: 3
    score_threshold: 0.6
    rerank: false

  compose:
    temperature: 0.7
    model: claude-haiku-4-5-20251001

  capsule_db: capsules_v1.json# 指向哪个capsule库版本

  每次跑 quality_runner.py --config experiment.yaml，结果写进
  eval/output/{experiment_id}/scores.csv。这样可以对比 exp_001 vs exp_002
  的weighted_total，autoresearch里这个数字就是 val_bpb。

  3. 中间过程日志（调试RAG必须）

  现在 conversations.csv 只记录最终对话。补一列：

  Ground_Nodes_Retrieved | Ground_Relevance_Score | Answer_Groundedness_Score

  让quality_runner.py 在每轮对话时把ground.py
  返回的节点ID也记录下来，scorer再对这列单独打分。

  做的是选择性加载capsule knowledge，不是向量检索——它是rule-based的条件触发（shou

  这意味着：
  - "调RAG
  invocation"实际是在问：什么情况下触发ground加载，加载哪个capsule的哪些节点
  - 传统RAGAS指标（context_recall/precision）假设有embedding检索，不完全适用

  你真正要调的超参数可能是：
  1. should_load_ground 的触发条件（现在是什么逻辑？）
  2. capsule里ground.nodes 的内容（哪些知识点写进去了）
  3. compose的prompt如何把ground注入回答
