# 基础讯息
capsule_name: 
要求：使用中文名称，简短清晰
格式：直接写就好，不需要双引号，例如：离婚经济生存指南
说明：这是UI界面显示的标题，要让人一眼知道这个胶囊是做什么的

status: draft:
要求：只能选择四个预设选项之一
格式：字符串： "draft" 、 "pending" 、 "approved"  、 "deprecated"（同样不需要双引号）
说明：新建时默认写 draft（草稿），专家或第三者审核通过后改为 approved（已启用）。小安只会加载 approved 状态的胶囊


# 触发条件
要求：收集3到5个典型的用户问句
格式：列表形式，每条用短横线和引号，例如 ["问题一", "问题二", "问题三"] 例子见下
说明：越口语化越好，覆盖用户最常问的情况。举例如下：
```
triggers:
  - "我离婚以后养不起自己和孩子怎么办"
  - "我没有收入不敢离婚"
  - "我怕离开他孩子怎么办"
```


# applicability 适用规则
```
说明：使用规则的本质是做场景验证 (Context Validation)。
应用目的：安全过滤与精准投放。解决的是「在这个特定情况下，给出这些建议是否安全、合适、有效？」的问题。它是第二道阀门，决定系统是否真的要执行这个知识库的逻辑。
判断逻辑：基于状态标记 (State Tags) 或 全局风险评估结果。需要结合多模态数据（情绪分析、危机检测、历史行为）。
    * 如果 use_when 条件满足 且 do_not_use_when 条件不满足→正式执行。
    * 否则→即使触发了，也要放弃或转接其他流程。
内容特点：抽象的状态描述或逻辑规则。通常是对用户当前处境的定性描述。
例子：
    * use_when: "用户表达经济恐惧" + "无即时人身危险"。
    * do_not_use_when: "用户正处于被暴力威胁中" + "手机被监控"。
```


use_when:
要求：每一点之间是“and”的关系
格式：列表形式，每项为一个描述性短语或标签（字符串）。
说明：这是正面清单。只有当所有列出的条件都符合时，才调用这个胶囊。
示例：
```
use_when:
  - 用户主要表达经济恐惧
  - 当前无即时人身危险  
  - 用户需要把大问题拆解为小步骤
```

do_not_use_when:
要求：每一点之间是“or“的关系
格式：列表形式，每项为一个否定性情境描述（字符串）。
说明：这是负面清单（红线）。只要触发其中一条，无论 use_when 是否匹配，都不调用这个胶囊。
示例：
```
do_not_use_when:
  - 用户正在遭遇暴力或当前不安全     
  - 用户手机/财务账户正被对方监控   
  - 用户有自伤风险或表达活不下去    
```


# recognize 认知引导
recognize.purpose: 
要求：用一句话精确概括
格式：单一字符串
说明：可以指导 小安 思考方向、语气
示例：
`recognize.purpose："确认用户的恐惧是合理的生存反应，建立共情连接。"`

conten
要求：撰写3到5条核心讯息要点
格式：列表形式，每条简短的一句话，例如 [" 暴力不只是“拳头” "]
说明：这是 小安 生成回答时的依据，不是最终回复原文。

# act 行动指引
act.purpose:
要求：用一句话精确概括
格式：单一字符串
说明：这里可以说明行动的重点、顺序重要性或者、行动背景（语境）重要性；可以没有针对性，跟其他胶囊的act.purpose一致。
示例：
`act.purpose: "引导用户将巨大的焦虑转化为可执行的小步骤。"`

steps:
要求：列出3到5个可执行的操作步骤/行动选项
格式：列表形式，每条用指令式写法，清晰明确
说明：避免模糊表述，比如"列一张最低生活清单：住处、吃饭、孩子上学、交通、医疗"这样的具体指导


scripts:
要求：提供1到2句话术模板
格式：列表形式，用引号包裹的句子
说明：教用户如何开口求助，降低启动门槛。例如"我现在还没决定离婚，但想了解保护自己和孩子需要准备什么"

safety_note:
要求：必填项，必须填写安全风险警示
格式：列表形式，每条一句话
说明：涉及人身安全、冲突、隐私、监控的风险的内容必须在此提示。例如"如果对方控制你的手机，操作要低调"

# ground 依据与限制
ground.purpose: 
要求：用一句话精确概括
格式：单一字符串
说明：这一句相当于写给 小安 的 System Prompt 片段，可以没有针对性，跟其他胶囊的ground.purpose一致。
示例：
`ground.purpose: "解释法律、实务依据与边界，管理用户对结果的期望。"`

legal_basis:
要求：列出相关法律法规的名称和条款范围
格式：列表形式，每条一条法律引用（字符串）
示例：
`legal_basis："《民法典》婚姻家庭编第1042条"`

practice_basis:
要求：列出 发布方 + 发布时间 + 实务指南/指导手冊名称（如果可以具体到章节）
格式：列表形式，每项为一条实务经验描述（字符串）
示例：
`practice_basis："源众于2026年发布的《未成年人家庭暴力强制报告实务手册》"`

limits:
要求：明确告知AI不能做什么，管理用户期望
格式：列表形式，每条一个限制说明
说明：防止过度承诺，例如"不能承诺离婚后一定能获得财产分割结果"
