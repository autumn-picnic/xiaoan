# 學術 PDF 批次轉 Markdown 管線設計

## 目標

將 `knowledge/source/論文_pdf/` 內的 81 篇學術 PDF 批次轉換為結構一致、閱讀順序正確的 Markdown，輸出到使用者明確授權建立的 `knowledge/source/論文_md/`。首輪必須完全離線運行；完成後以品質報告判斷是否需要用低成本 API 修復低信心結果。

原始 PDF 是不可變來源。程式不得修改、移動、重新命名或覆蓋任何 PDF。

## 成功標準

- 每個可讀 PDF 都產生一個同名 `.md` 文件。
- 自動區分有文字層的頁面與需要 OCR 的頁面；OCR 使用簡體中文和英文。
- 單欄、雙欄、跨欄標題及常見側欄按人類閱讀順序輸出。
- 刪除重複頁頭、頁尾、頁碼和頁底腳註，但保留正文引用標號與文末參考文獻。
- 每篇輸出標題、作者、作者機構、期刊、發表時間、摘要、正文和文獻八個固定部分。
- 缺失或無法可靠識別的 metadata 不得臆造。
- 單篇失敗不阻塞批次；所有異常、信心分數和處理方式寫入機器可讀報告。
- API 是可選的後處理能力；沒有 API key 時，離線流程仍完整可用。

## 方案選擇

### 採用：離線座標分析，再按信心選擇 API 修復

首輪使用已安裝的 PyMuPDF、pdfplumber、Pillow、Tesseract 和 Poppler 工具。這套工具能保留文字與頁面座標，適合針對目前以中文學術期刊為主的集合建立可解釋、可測試的版面規則。

離線首輪完成後，只對低信心文件測試本機大型版面模型或低成本 API。API 優先接收首頁文字、版面候選和待修復欄位，不預設上傳全文或整份 PDF。

### 未作為首選：Docling 或 Marker

本機大型版面模型可能提升特殊版式效果，但模型和依賴較大、首輪安裝及運行成本較高，且不能直接保證學術 metadata 正確。它們可作為低信心樣本的比較基線，而不是 81 篇的預設引擎。

### 未採用：雲端優先

雲端視覺或語言模型可能改善特殊頁面，但會增加成本、網絡依賴和資料外傳。它不符合先完成離線首跑的要求。

## 專案結構

```text
scripts/
  convert_papers_to_markdown.py       # CLI 與批次協調
  paper_converter/
    __init__.py
    models.py                         # Page、Block、PaperMetadata、QualityReport
    preflight.py                      # PDF 與逐頁文字層檢測
    extraction.py                     # 原生文字塊提取
    ocr.py                            # 頁面渲染與 Tesseract OCR
    layout.py                         # 欄位辨識與閱讀順序重建
    cleanup.py                        # 頁頭、頁尾、頁碼、腳註清除
    structure.py                      # metadata、摘要、正文、文獻切分
    render.py                         # Markdown 渲染
    quality.py                        # 信心評分與告警

knowledge/source/
  論文_pdf/                           # 不可變輸入
  論文_md/                            # 生成的 Markdown 與批次報告
```

每個模組有單一責任，模組間傳遞具名資料模型，避免把 PDF 讀取、版面推斷和 Markdown 格式化耦合在同一個大函數中。

## CLI

預定提供以下操作：

```bash
# 轉換預設來源目錄中的全部 PDF
python scripts/convert_papers_to_markdown.py

# 只轉換一篇
python scripts/convert_papers_to_markdown.py --input "knowledge/source/論文_pdf/示例.pdf"

# 覆蓋已存在的成功輸出
python scripts/convert_papers_to_markdown.py --overwrite

# 只更新品質報告，不寫 Markdown
python scripts/convert_papers_to_markdown.py --report-only

# 設計驗證後才可能加入的可選 API 修復
python scripts/convert_papers_to_markdown.py --api-refine
```

預設跳過已有成功輸出的文件。輸入、輸出路徑可以由 CLI 覆蓋，但預設值固定為本設計中的兩個目錄。

## 處理流程

### 1. 預檢與逐頁路由

程式讀取 PDF 頁數、尺寸、旋轉資訊及文字塊。每頁計算：

- 可提取字元數；
- 有效中英文字符比例；
- 文字塊數量；
- 文字覆蓋區域；
- 亂碼及不可見字符比例。

達到文字品質門檻的頁面使用原生文字層；低於門檻的頁面進入 OCR。判斷以頁為單位，允許同一 PDF 同時包含原生文字頁和掃描頁。

### 2. OCR fallback

需要 OCR 的頁面以 300 DPI 渲染，根據 PDF 旋轉資訊校正方向，再調用 Tesseract 的 `chi_sim+eng`。OCR 必須輸出單詞或行的 bounding box，而不是只有純文字，讓後續版面模組可以與原生文字頁共用同一資料模型。

OCR 失敗時保留頁面錯誤，不以空白結果冒充成功。批次繼續處理其他頁面和文件。

### 3. 版面辨識與閱讀順序

所有文字塊先正規化為頁面比例座標。版面模組分析正文區內的水平空隙、文字塊 X 軸分布和行寬，將頁面分類為：

- 單欄；
- 雙欄；
- 跨欄標題加單欄；
- 跨欄標題加雙欄；
- 含側欄的複雜版面。

跨越主要欄界的短大字塊視為跨欄候選。正文塊按欄由左至右、欄內由上至下排列。側欄若位於首頁且內容與摘要形態相符，可作為摘要候選；否則標為低信心，不直接插入正文中間。

### 4. 頁頭、頁尾、頁碼與腳註清理

清理分兩階段：

1. 跨頁規則：正規化文字中的頁碼和空白後，找出在多頁頂部或底部重複出現的短行，辨識期刊名、卷期、文章短標題及頁碼。
2. 單頁規則：結合垂直位置、字號、分隔線、腳註標記和正文基線，辨識頁底註釋區。

腳註區文字從正文中刪除；正文內的上標或圈號引用標記保留。文末以「參考文獻」「文獻」「References」等標題開始的獨立部分不按腳註刪除。

若規則無法區分頁底腳註與正文，寧可保留文字並給出 `possible_footnote_leak` 告警，避免靜默刪除正文。

### 5. 學術結構與 metadata 提取

提取器主要使用首頁和文末，來源優先級如下：

1. 首頁版面、明確標籤及字體特徵；
2. PDF 文件名中的標題和作者；
3. PDF metadata；
4. 期刊頁頭、卷期和日期模式。

標題使用首頁頂部、較大字號、跨欄且與文件名相似的候選。作者由標題下方短行及文件名後綴交叉驗證。作者機構由作者簡介、帶括號的單位行、院校／研究機構關鍵詞提取。期刊和日期由首頁頁頭、卷期及中英文月份模式提取。

摘要從「摘要」「内容摘要」「Abstract」開始，到「關鍵詞」「关键词」「Key words」或第一個正文標題前結束。正文在摘要後開始，在獨立文獻標題前結束。文獻從獨立文獻標題到文件結束。

若原文沒有摘要或文末參考文獻，輸出 `原文未載`；若原文可能有但提取器無法可靠定位，輸出 `未識別` 並降低信心。這兩種情況必須區分。

### 6. Markdown 渲染

每篇文件使用固定骨架：

```markdown
# 標題

## 作者

作者姓名

## 作者機構

機構名稱或「未識別」

## 期刊

期刊名稱或「未識別」

## 發表時間

YYYY-MM、YYYY 或「未識別」

## 摘要

摘要內容、「原文未載」或「未識別」

## 正文

按閱讀順序排列並保留層級的正文

## 文獻

文末參考文獻、「原文未載」或「未識別」
```

正文標題在信心足夠時轉成 `###` 以下層級。普通段落合併 PDF 中因換行產生的斷行；列表、中文序號、法律條文編號和文獻條目保持分行。

## 品質報告

批次完成後在 `knowledge/source/論文_md/_conversion_report.json` 寫入：

- 輸入和輸出路徑；
- 頁數、原生文字頁數和 OCR 頁數；
- 每頁版面分類；
- 八個輸出部分的存在狀態；
- metadata 信心；
- 版面／閱讀順序信心；
- 正文完整度信心；
- 總分；
- 告警、錯誤及耗時；
- 程式與規則版本。

信心分數由可觀察信號計算，不由模型憑空生成。任何硬錯誤、正文過短、頁面遺失、欄位缺失或版面無法分類都會降低總分。報告須能列出需要人工或 API 複核的文件。

## 錯誤處理與可恢復性

- 單篇文件使用獨立的處理邊界；異常記錄後繼續下一篇。
- 輸出先寫到臨時文件，成功完成並驗證非空後再原子替換目標 Markdown，避免留下半成品。
- 預設跳過已有且報告標記成功的文件；`--overwrite` 明確要求重建。
- 缺少 Tesseract 語言包時，只影響需要 OCR 的頁面，錯誤訊息須給出所缺語言。
- 依賴缺失、加密 PDF、損壞頁面和超時使用不同錯誤碼。
- 日誌不得包含 API key 或其他環境機密。

## 測試策略

### 單元測試

- 單欄、雙欄、跨欄標題和側欄的合成文字塊排序；
- 跨頁重複頁頭／頁尾辨識；
- 頁底腳註與文末參考文獻區分；
- 中文標題、作者、機構、期刊、日期和摘要邊界；
- `原文未載` 與 `未識別` 的不同輸出；
- Markdown 固定結構和原子寫入。

### 實際樣本驗證

先建立至少四類代表樣本：

- 常規單欄；
- 常規雙欄；
- 側欄或複雜雙欄；
- 最低文字密度或人工製造的 OCR fallback 頁。

對樣本人工核對閱讀順序、八個輸出部分、正文開頭／結尾、腳註清除和文獻保留。樣本通過後全量運行 81 篇，再按總分和告警抽查高、中、低信心結果。

## API 必要性評估

離線首跑後，從最低信心及不同錯誤類型中選取代表文件。API 測試優先只提交：

- 首頁提取文字；
- 文字塊及相對座標的精簡表示；
- 離線候選 metadata；
- 明確待修復的欄位或段落邊界。

不預設提交全文或 PDF。使用支持結構化輸出的低成本模型，要求返回固定 JSON schema。比較指標包括：

- metadata 欄位準確率；
- 摘要、正文、文獻邊界；
- 閱讀順序修復率；
- 單篇與全量估算成本；
- 延遲和失敗率。

只有當 API 對低信心樣本產生明顯、可重複的提升，才實作可選的 `--api-refine`。API key 僅從環境變數讀取。未配置 key、API 不可用或修復失敗時，保留離線輸出和告警，不影響批次完成。

## 範圍邊界

本次工作只建立 PDF 到 Markdown 的轉換管線、測試、離線輸出和 API 必要性評估。它不把論文內容寫入 legal mechanism wiki，不更新 `knowledge/index.md`、`knowledge/log.md` 或 source registry，也不對論文內容進行法律解釋。
